## AR-006: Use Canonical Internal Identifiers And Separate UI Localization

### Status
  
Implemented

### Context
  
Warband Manager currently parses character export files generated by World of Warcraft.

Many exported values are localized by the game client language. Examples include:
- Currencies
- Reputations
- Character Statistics
- Equipment-related Information

As a result, the current parser is primarily compatible with English character exports.

The application itself is also currently English-only. User-visible elements such as:
- Labels
- Buttons
- Tab Names
- Table Headers
- Dialog Text

are displayed in English.

Supporting additional languages requires solving two separate problems:

1. Parsing localized World of Warcraft exports.
2. Displaying the application user interface in different languages.

A solution is needed that remains simple, maintainable, and consistent with the project's existing principles.

### Decision
  
The application separates **data localization** from **UI localization**.

#### Parsing Strategy

The application maintains a language-independent internal data model.

Localized values found in character exports are translated into canonical internal identifiers during parsing.

Example:

```text
English Export:
Gold

German Export:
Gold

French Export:
Or
```

All of the above are mapped internally to:

```text
gold
```

After parsing, application components should operate on canonical identifiers rather than localized text whenever practical.

Storage should persist canonical identifiers instead of language-specific values.

#### UI Localization Strategy

User interface text is localized independently from the parser.

User-facing strings should be referenced through a localization layer rather than hardcoded directly into widgets.

Examples include:
- Labels
- Buttons
- Dialogs
- Table Headers
- Tab Titles
- Status Messages

The displayed language is determined by application settings.

#### Settings

The application maintains separate settings for parsing and display.

##### Export Language

Determines how World of Warcraft exports are interpreted during parsing.

Examples:

```text
English
German
French
```

##### Application Language

Determines which language is used for the user interface.

Examples:

```text
English
German
French
```

These settings are intentionally independent.

This allows scenarios such as:

```text
WoW Export Language: German
Application Language: English
```

or

```text
WoW Export Language: English
Application Language: French
```

### Consequences

#### Positive
- Parsing logic becomes language-independent after import
- Storage format remains consistent regardless of export language
- UI localization and parser localization can evolve independently
- New languages can be added with minimal impact on the existing architecture
- Character files remain portable across language settings
- Supports players using different WoW client languages
- Aligns with the project's simplicity and local-first principles

#### Negative
- Translation mappings must be maintained for every supported language
- Additional localization resources are required
- Existing parser components may require refactoring to use canonical identifiers consistently
- UI text must be moved away from hardcoded strings over time

### Notes
  
This decision does not mandate a specific localization technology.

The project may use:
- Translation dictionaries
- Qt translation mechanisms
- Another lightweight localization approach

provided that:

1. Parsing localization and UI localization remain separate concerns.
2. Canonical internal identifiers remain the source of truth after parsing.
